# 프로필 사진 추가 후 홈이 느려지는 이유

## 1. 홈이 “느리다”고 느껴지는 지점

홈 화면은 **진입 시** 다음 순서로 동작합니다.

1. `initState` → `_runInitialLoad()` 호출  
2. **`await _loadAll()`** → 아래 세 API를 **모두 끝날 때까지 대기**
   - 오늘의 말씀 (Daily Verse)
   - 기도 목록 (Prayers, 최대 5개)
   - 감사 (Gratitude: 오늘 일기 + 피드 + 스트릭)
3. 세 개 **전부** 완료된 뒤에만 `_initialLoadDone = true` → 이때서야 본문(스크롤/카드) 표시

그래서 **“홈이 느리다”** = “로딩 스피너가 오래 돌고, 본문이 늦게 뜬다”로 체감됩니다.

---

## 2. 프로필 사진이 관여하는 부분

### 2-1. API 응답 크기

- 기도 목록 API(`GET /api/prayers`)는 **각 기도마다** `user` 정보를 붙여 보냅니다.
- `user` 안에 `profile_image_url`(Supabase Storage URL)이 포함됩니다.
- URL 하나가 보통 150~250자 정도라, 기도 5개면 **약 1~2KB** 정도 응답이 늘어납니다.
- 이 정도 크기만으로는 “엉청 느려진다” 수준이 되기 어렵고, **주된 원인**이라고 보기는 힘듭니다.

### 2-2. 이미지 로딩(CachedNetworkImage)

- 본문이 **한번 뜬 다음**에는 기도 카드마다 `CachedNetworkImage`가 프로필 사진을 **비동기**로 받아옵니다.
- 이건 **본문이 뜨기 전**을 막지 않습니다. (이미지 로딩이 끝나기 전에는 placeholder로 첫 글자만 보여줌)
- 따라서 **“첫 화면이 늦게 뜨는”** 현상의 **직접 원인**은 아닙니다.

### 2-3. 실제로 느려질 수 있는 경우

프로필 사진 **추가 후** 홈이 느려졌다면, 아래 조합일 가능성이 큽니다.

1. **홈을 “전부 로드된 뒤에만” 보여주는 구조**
   - 말씀 / 기도 / 감사 **세 API 중 하나만** 느려져도, **전체 홈 표시가 그만큼 지연**됩니다.
   - 예: 기도 API가 3초 걸리면, 말씀·감사가 0.5초에 끝나도 사용자는 3초 동안 로딩만 봅니다.

2. **Railway/백엔드·DB 상태**
   - 콜드 스타트, DB 부하, Supabase 응답 지연이 있으면 기도 API가 예전보다 느려질 수 있습니다.
   - 프로필 사진을 쓰면서 **Storage 사용**이 늘어나면, 같은 시점에 다른 API가 느려졌을 수도 있습니다.

3. **네트워크**
   - 사용자 환경에서 한동안 느린 네트워크였을 수 있고, 그때 “프로필 추가 후”로 기억에 남았을 수 있습니다.

정리하면, **“프로필 사진 URL이 API에 포함 + 홈을 세 API 모두 끝날 때까지 숨겨 두는 구조”** 때문에,  
API가 조금만 느려져도 **“홈이 엉청 늦게 뜬다”**고 느끼기 쉽습니다.  
프로필 사진 **이미지 다운로드** 자체보다는, **“언제 본문을 보여주느냐”**가 더 큰 요인입니다.

---

## 3. 개선 방향 (요약)

- **지금**: 세 API 전부 완료 → 그다음에 홈 본문 표시 → 체감 “홈이 느리다”.
- **개선**:  
  - **첫 진입**에는 API 완료를 기다리지 않고, **먼저 본문(레이아웃)** 을 보여 주고,  
  - 말씀/기도/감사는 **백그라운드에서 로드**되면 그때그때 채워 넣기.

이렇게 바꾸면 “프로필 사진 추가 후 홈이 늦어졌다”는 체감을 줄일 수 있고,  
이미지 로딩은 여전히 비동기라서 화면이 뜨는 시점에는 영향을 거의 주지 않습니다.
